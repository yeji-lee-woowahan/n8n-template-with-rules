# Virtual Verify 스킬

> 코드와 기획서를 대조하여 AI가 가상으로 동작을 검증한다.
> 실제 브라우저 테스트 전에 논리적 오류와 누락을 사전 탐지한다.

## 사용 시점

- 구현 완료 후, 코드리뷰 전
- 커맨드 `5.가상검증`에서 호출

## 프로세스

> 모든 명령어는 `n8n-template/` 폴더에서 실행한다.

### 1. 변경 범위 파악

```bash
git diff --name-only
```

- 변경된 파일이 없으면 "검증할 변경사항이 없습니다" 안내 후 **중단**
- 변경된 파일 목록에서 관련 기능을 파악한다

### 2. 관련 기획서 식별

- 변경 파일이 속한 기능의 기획서(`n8n-template/docs/기획서/`)를 찾아 읽는다
- 기획서의 **"필수 확인 동작" 섹션**이 시나리오의 유일한 소스

### 3. 기획서-코드 불일치 체크

코드 추적 중 기획서와 구현이 다른 부분을 발견하면:
1. 사용자에게 불일치 내용을 알림
2. 사용자 컨펌 후 기획서를 수정 (기획서가 시나리오 소스이므로 항상 최신 유지 필수)
3. 수정 완료 후 다음 단계 진행

### 4. 유즈케이스 변환

기획서 동작 섹션을 유즈케이스로 변환:
- **일반 케이스**: 정상 동작 흐름
- **예외 케이스**: 에러, 엣지케이스, 권한, 빈 상태, 타이밍 등

### 5. 코드 추적 검증

각 유즈케이스별로 코드를 추적:

1. **코드 추적**: 해당 동작이 실행되는 코드 경로를 따라간다
2. **상태 변화 확인**: 각 단계에서 상태가 어떻게 변하는지 추적한다
3. **렌더링 결과 추론**: 상태 변화에 따른 UI 출력을 예측한다
4. **판정**: 기획서 기대 동작과 코드 동작이 일치하는지 판단

```
✅ 시나리오: [시나리오명]
   - 코드 경로: ComponentA → useHookB → apiCall → 상태 업데이트
   - 기대 동작: [기획서 설명]
   - 코드 동작: [추적 결과]
   - 판정: 일치

❌ 시나리오: [시나리오명]
   - 코드 경로: ComponentA → useHookB
   - 기대 동작: [기획서 설명]
   - 코드 동작: [추적 결과 - 불일치]
   - 원인: [왜 다른지]
```

### 6. 예외 케이스 기획서 반영

검증 중 발견된 예외 케이스(기획서에 없던 것)를:
1. 사용자에게 보고
2. 컨펌 후 기획서 동작 섹션에 추가
3. 다음 가상검증 시 자동으로 검증 대상에 포함됨

### 7. 자동화 테스트 추천

검증 중 "코드 실행 없이는 확인 불가"한 케이스가 있으면 `.agent/개발진행/테스트추천.md`에 기록한다. 없으면 스킵.

추천 기준:
- 가상검증으로 잡기 어려운 엣지케이스
- 터지면 장애로 이어지는 것
- 복잡한 데이터 변환/파싱

### 8. 자동화 테스트 실행 (해당 시)

```bash
pnpm test:run
```

- 테스트가 있으면 실행하고 결과 확인
- 실패한 테스트가 있으면 원인 분석

### 9. 결과 정리

#### 모두 통과

> "가상검증 완료. 모든 시나리오가 통과했습니다.
> **6.코드리뷰** 커맨드를 실행해주세요."

#### 이슈 발견

1. 발견된 이슈들을 정리한다 (시나리오별 불일치 사항)
2. `.agent/개발진행/구현계획.md`에 수정 세션을 추가한다:
   - 기존 세션은 건드리지 않음
   - 맨 아래에 수정할 내용 명시
3. 사용자에게 수정 진행을 요청한다

> "가상검증에서 이슈가 발견되었습니다.
> 구현계획.md에 수정 세션을 추가했습니다.
> **4.구현** 커맨드로 수정 후 다시 **5.가상검증**을 실행해주세요."
