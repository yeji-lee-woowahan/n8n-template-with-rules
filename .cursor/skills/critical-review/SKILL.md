# Critical Review 스킬

> 변경된 코드와 기획서를 보며 "이게 정말 맞나?"를 비판적으로 검토한다.
> 칭찬보다는 문제점을 찾는 데 집중하고, 놓치기 쉬운 엣지 케이스를 짚어낸다.

## 사용 시점

- 코드리뷰 완료 후
- 커맨드 `7.비판적검토`에서 호출

## 프로세스

> 모든 명령어는 `n8n-template/` 폴더에서 실행한다.

### 1. 현황 파악

```bash
git diff --name-only
```

- 변경된 파일이 없으면 "검토할 변경사항이 없습니다" 안내 후 **중단**
- 변경된 코드와 연관된 기획서를 찾아 읽는다

### 2. 비판적 검토

**체크박스를 채우려 하지 말고**, 다음 관점에서 자유롭게 비평한다.

#### 상태 추적 보고서

- 관련 상태가 어디서 초기화되고 어디서 업데이트되는지 보고서 작성 → 단일 소스 원칙 확인
- 새로 만든 상태/함수를 나열하고 기존 스토어/컴포넌트/유틸과 중복이 없는지 확인
- 상태를 나열하고 각 상태를 읽거나 쓰는 컴포넌트의 depth가 적절한지 검토

#### 깨지기 쉬운 시나리오 (각 3개 이상)

- **상태 동기화** 어긋남 시나리오
- **타이밍/순서** 어긋남 시나리오
- **사용자 행동** 관점에서 깨지기 쉬운 시나리오

#### 확장성 검토

- 구현 코드가 확장되는 시나리오를 작성 → 확장에 열려있는가?
- 구현 내용이 수정되는 시나리오를 작성 → 수정에 닫혀있는가?

#### 암묵적 계약 탐지

- **초기화 의존**: 컴포넌트/훅이 정상 동작하려면 부모(또는 특정 페이지)가 선행 호출해야 하는 초기화 로직(sync 훅, fetch 호출 등)이 있는지 확인. 그 규칙이 타입이나 런타임 에러로 강제되는지 확인.
- **Store 선점 가정**: 전역 스토어나 Context에서 값을 읽는 컴포넌트가 해당 값이 반드시 채워져 있다고 가정하는지 확인. 새로고침/직접 URL 접근 시에도 안전한지 점검.
- **탐색 경로 의존 상태**: 탭 이동/페이지 내 네비게이션으로만 테스트하면 이전 상태가 남아 정상 동작하지만, 새로고침/딥링크 접근 시 깨지는 경로가 있는지 확인.
- **타입 갭**: 타입 상 null/undefined가 허용되지만 실제로는 non-null이어야 정상 동작하는 곳이 있는지 확인. optional chaining으로 조용히 실패하는 패턴 포함.

#### 추가 확인

- 500줄이 넘는 파일이 있는가?

발견한 문제점과 개선 방향을 자유롭게 서술한 다음 사용자에게 피드백을 받는다.

### 3. 자동화 테스트 추천 (선택)

깨지기 쉬운 시나리오 중 자동화 테스트로 잡아야 할 것이 있으면:
1. `.agent/개발진행/테스트추천.md`에 추가 기록
2. 없으면 스킵

### 4. 검토 결과 및 다음 단계

#### 문제 없음

> "비판적 검토 완료. 문제 없습니다.
> **8.문서정리** 커맨드를 실행하여 마무리해주세요."

#### 수정/리팩토링 필요

1. 발견된 문제점과 수정 방향을 정리한다
2. `.agent/개발진행/구현계획.md`를 업데이트한다:
   - 기존 세션은 건드리지 않음
   - 맨 아래에 **수정 세션**을 추가
3. 사용자에게 수정 진행을 요청한다

> "수정이 필요합니다. 구현계획.md에 수정 세션을 추가했습니다.
> **4.구현** 커맨드로 수정을 진행해주세요."
