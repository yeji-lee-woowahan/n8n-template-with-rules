# 컴포넌트 설계

## 네이밍 규칙

### 피해야 할 단어

| 금지 | 이유 |
|------|------|
| `data`, `info`, `item` | 너무 일반적, 의미 불명확 |
| `val`, `res`, `tmp` | 축약형, 가독성 저하 |

### 동사 구분

| 동사 | 용도 |
|------|------|
| `get` | 동기적, 가벼운 연산 |
| `fetch` | 네트워크 요청 |
| `compute` / `calculate` | 무거운 연산 |

### 함수/핸들러 네이밍

| 규칙 | Bad | Good |
|------|-----|------|
| 목적어 포함 | `handleClick` | `handleLoginButtonClick` |
| 단위 명시 | `timeout`, `size` | `timeoutMs`, `sizeMb` |

### 불리언 네이밍

| 규칙 | Bad | Good |
|------|-----|------|
| 접두사 사용 | `loading`, `visible` | `isLoading`, `isVisible` |
| 긍정문 사용 | `isDisabled`, `isNotValid` | `isEnabled`, `isValid` |

접두사: `is`, `has`, `can`, `should`

## 컴포넌트 분리 전략

### 기본 전략: 한 파일 = 로직 + UI

- 비즈니스 로직(Hook 호출)과 JSX(UI)를 **한 컴포넌트 안에** 작성
- **처음부터 Container/Presenter로 나누지 않는다** (오버엔지니어링)

### 분리가 필요한 경우

다음 중 **하나 이상 충족**해야 분리:

| 조건 | 설명 |
|------|------|
| **순수 재사용** | 비즈니스 로직은 다르지만, 똑같은 디자인이 다른 곳에서 필요 |
| **공통 UI 라이브러리** | 도메인 맥락 없이 프로젝트 전반에서 사용 (Button, Card 등) |

### 복잡도 기반 분리 (Soft Limit 500줄)

단순히 라인 수가 길다고 분리하지 않는다. (Types, 스타일 등으로 길어질 수 있음)

**500줄이 넘어가면** 다음 순서로 체크하고 리팩토링 고려:

1. 상수/타입 정의가 내부를 차지하는가? → `const.ts`, `types.ts`로 추출
2. useEffect나 상태 로직이 복잡한가? → Custom Hook으로 추출
3. 그래도 JSX가 너무 긴가? → 그때 하위 컴포넌트로 분리

### 분리 금지 케이스

| 케이스 | 이유 |
|--------|------|
| 1회성 복잡 UI | 재사용 안 되면 같은 파일에 유지 |
| 단순 Wrapper | 불필요한 추상화 |
| 한 곳에서만 쓰이는데 Container/View 분리 | 오버엔지니어링 |

## UI 컴포넌트 vs 도메인 컴포넌트

### 정의

| 구분 | 위치 | API 호출 | 역할 |
|------|------|----------|------|
| **UI 컴포넌트** | `shared/ui/` | ❌ 금지 | Props만 받아서 렌더링 (순수) |
| **도메인 컴포넌트** | `features/{domain}/components/` | ✅ 가능 | API 호출 + UI 컴포넌트 조합 |

### 판단 기준

1. "이 컴포넌트가 API를 호출해야 하는가?" → Yes면 도메인 컴포넌트
2. "도메인 맥락(Mcp, MySpace 등)이 있는가?" → Yes면 도메인 컴포넌트
3. 둘 다 No면 → UI 컴포넌트 (`shared/ui/`)

## 커스텀 훅 설계

### 반환값 규칙

| 반환값 개수 | 방식 |
|------------|------|
| 1-2개 | 튜플 `[value, setValue]` 또는 단일 값 |
| 3개 이상 | 객체 `{ data, isLoading, actions }` |

```typescript
// 3개 이상 반환 시
const { data, isLoading, refetch } = useMcpData(mcpId);
```

### 인자 규칙

| 인자 개수 | 방식 |
|----------|------|
| 1-2개 | 개별 인자 `(id, options)` |
| 3개 이상 | 객체 `({ id, filter, page })` |

```typescript
// 3개 이상 인자 시
const useMcpList = ({ filter, page, sort }: McpListParams) => {
  // ...
};
```

### Headless 원칙

- 훅에서 **JSX 반환 금지**
- UI 제어권은 **컴포넌트에 위임**

```typescript
// Bad - 훅에서 JSX 반환
const useModal = () => {
  return <Modal>...</Modal>;  // X
};

// Good - 상태만 반환
const useModal = () => {
  const [isOpen, setIsOpen] = useState(false);
  return { isOpen, open: () => setIsOpen(true), close: () => setIsOpen(false) };
};
```

### 훅 조합 원칙

거대한 `usePageLogic` 대신 작은 훅들을 조합:

```typescript
// Bad
const useMcpPageLogic = () => {
  // 300줄의 모든 로직...
};

// Good
const McpListPage = () => {
  const { mcpList, isLoading } = useMcpList(filter);
  const { search, setSearch } = useMcpSearch();
  const { filters, setFilter } = useMcpFilters();
  // ...
};
```

## FSD 레이어 배치

### 새 컴포넌트/훅/유틸 배치 기준

| 질문 | 배치 위치 |
|------|----------|
| 앱 전역에서 쓰이는 공용 컴포넌트? | `shared/ui/` |
| 특정 도메인 기능? | `features/{domain}/` |
| 페이지 진입점? | `pages/` |
| 앱 설정/프로바이더? | `app/` |

### features/ 분류 기준

**"무엇을 다루는가"** 기준 (사용되는 곳 기준 X)

```
// MySpace 페이지에서 쓰는 MCP 관련 모달
features/mcp/modals/McpDetailModal.tsx    // O
features/myspace/modals/McpDetailModal.tsx // X
```
