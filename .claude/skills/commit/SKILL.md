# Commit 스킬

> 커밋 히스토리를 "나중에 봐도 이해되는 기록"으로 남긴다.
> 무엇을 왜 바꿨는지를 명확하게 전달하는 커밋 메시지를 작성한다.

## 사용 시점

- 코드리뷰/비판적검토/문서정리 완료 후 최종 커밋
- 커맨드 `9.커밋`에서 호출

## 프로세스

> 모든 명령어는 `n8n-template/` 폴더에서 실행한다.

### 1. Stage 상태 확인

```bash
git diff --staged --name-only
```

- **staged된 파일이 없으면**: "커밋할 파일을 stage 해주세요" 안내 후 **중단**
- **staged된 파일이 있으면**: 다음 단계로 진행

### 2. 코드 검증 실행

```bash
pnpm verify
```

ESLint + Prettier + TypeScript 타입 체크. 통과한 경우에만 다음 단계로 진행.

### 3. 변경 유형 판단

`git diff --staged --name-only` 결과 분석:

| 유형 | 조건 |
|------|------|
| 문서 수정 | `docs/**/*.md` 파일만 변경 |
| 코드 수정 | `docs/` 외 파일만 변경 |
| 혼합 | 둘 다 변경 |

### 4. 변경 유형별 체크리스트

#### 문서 수정

- [ ] `n8n-template/docs/기획서/0-기획서-작성가이드.md` 규칙에 맞춰 양식을 다듬었다

#### 코드 수정

- [ ] 명세서에 수정 내용을 반영하였다
- [ ] 코드리뷰를 진행하였다

#### 혼합

위 모든 체크리스트를 보여준다.

사용자에게 질문:

> 체크리스트를 모두 완료했나요? 남은 작업이 있나요?

- "완료했다" / "없다" → 다음 단계
- "남은 작업이 있다" → 해당 작업 완료 안내 후 **중단**

### 5. Jira 티켓 정보 추출

입력받은 Jira 티켓 URL에서 이슈 키를 파싱한다.

`jira_get_issue` MCP 도구를 사용하여 티켓 정보(key, summary)를 가져온다.

**티켓 URL이 없거나 MCP 호출 실패 시:**

> 1. 티켓 URL 다시 입력
> 2. 티켓 정보 직접 입력 (프로젝트명-이슈번호, 티켓 제목)
> 3. 티켓 정보 없이 커밋 메시지 직접 작성

### 6. 커밋 메시지 작성 및 확인

staged된 변경 내용(`git diff --staged`)을 분석하여 작성:

```
[프로젝트이름-이슈번호] 수정 한 줄 요약

- 변경사항 상세 요약
```

**작성 원칙: "무엇을 왜" (What & Why)**

- **제목**: 변경의 목적/결과를 간결하게 (50자 이내 권장)
- **본문**: 해결된 문제, 추가된 기능, 변경 이유를 bullet point로 요약

**피해야 할 것: 구현 상세 (How)**

- 코드 diff에서 바로 보이는 내용은 생략
- ❌ "resetKey prop 추가", "lastSyncedAt을 의존성에 추가"
- ✅ "동기화 후 UI가 갱신되지 않던 문제 해결"

작성한 커밋 메시지를 사용자에게 보여주고 확인:

> 이 커밋 메시지로 진행할까요?

### 7. 커밋 진행

staged된 파일만 커밋. 멀티라인 메시지는 heredoc 사용:

```bash
git commit -m "$(cat <<'EOF'
[프로젝트이름-이슈번호] 수정 한 줄 요약

- 변경사항 상세 요약
EOF
)"
```
