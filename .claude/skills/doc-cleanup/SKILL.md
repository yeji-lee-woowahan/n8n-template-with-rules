# Doc Cleanup 스킬

> 코드만 봐서는 알기 어려운 "왜 이렇게 했는지"를 코드 주석으로 남긴다.
> 별도 설계문서 대신, [CORE] 주석으로 설계 의도를 코드에 직접 기록한다.

## 사용 시점

- 비판적 검토 완료 후
- 커맨드 `8.문서정리`에서 호출

## 프로세스

### 1. 주석 보강

변경 코드에 [CORE] 주석이 필요한 위치를 확인한다.

**절차:**

1. `git diff --name-only HEAD`로 변경 파일 확인
2. 변경 코드에서 [CORE] 주석 후보 위치를 3레벨 기준으로 나열
   - 레벨 1: 모듈/훅 정의부 (핵심 책임, 의존 관계, 호출자)
   - 레벨 2: 다단계 흐름 함수 (API→상태→UI 같은 cross-cutting 흐름 요약)
   - 레벨 3: 설계 결정 분기점 (코드만 봐서는 '왜'를 알기 어려운 곳)
3. **사용자에게 진입점 후보 목록을 제시하고 컨펌 받기**
4. 승인된 위치에 [CORE] 주석 작성

**참고:** [주석-가이드.md](./주석-가이드.md)

### 2. 주석 관리

**점검 항목:**

- `[SPEC]` - 기획 요구사항: 기획서 링크/근거 있는지
- `[CORE]` - 기술적 의사결정: 이유가 있는지
- `[TODO]` - 기술 부채: 표시되어 있는지
- `[WARN]` - Side Effect: 영향 범위 적혀있는지

**원칙:**

- 코드는 What, 주석은 Why
- 좋은 이름이 주석보다 낫다

### 3. 기획서 동기화

`.agent/개발진행/구현계획.md` 참고하여 기획서(`n8n-template/docs/기획서/`)-코드 불일치를 확인하고 **실제 수정**까지 수행:
구현계획이 없는 경우에는 코드 기반으로 확인한다.

**확인 항목:**

- 구현된 기능이 기획서에 반영되어 있는지 (코드에 구현되어 있는데 기획서 동작에 없는 경우 꼭 체크)
- 기획서에 명시된 내용이 모두 구현되었는지
- 구현 중 변경된 사항이 기획서에 업데이트되었는지
- 필수 확인 동작 섹션이 최신 상태인지

**불일치 발견 시:**

1. 사용자에게 불일치 내용을 보고
2. 사용자 컨펌 후 `n8n-template/docs/기획서/0-기획서-작성가이드.md` 참고해서 기획서를 직접 수정
3. 수정 결과를 사용자에게 확인

### 4. 자동화 테스트 판단 (선택)

`.agent/개발진행/테스트추천.md`를 확인한다.

- 추천 목록이 있으면: 사용자에게 목록을 보여주고 각 항목별로 "작성 / 스킵" 판단을 받는다
- 추천 목록이 없으면: 스킵

### 5. 작업 문서 정리

작업이 완료되면 아래 문서들을 비워달라고 사용자에게 요청한다 (각 파일의 작성 양식만 남김):

- `.agent/개발진행/구현계획.md`
- `.agent/개발진행/테스트추천.md` (내용이 있는 경우)

> "문서정리 완료.
> **9.커밋** 커맨드를 실행하여 커밋해주세요."
