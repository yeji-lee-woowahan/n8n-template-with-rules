# 타입 전략

## 기본 원칙

| 원칙 | 설명 |
|------|------|
| orval 생성 타입 사용 | API 응답 타입 직접 정의 금지 |
| 기존 타입 활용 | `Pick`, `Omit`, `Partial` 등으로 변형 |
| 타입 좁히기 | `any`, `as`, `!` 대신 Type Guard 사용 |

---

## 단일 소스 원칙 (타입을 정의하지 말고 추출하라)

> 타입을 직접 정의하면 원본과 동기화가 깨진다. **항상 기존 타입에서 파생**하라.

### 체크리스트

- [ ] API 요청/응답 타입을 직접 정의하지 않는다 (orval 생성 타입 사용)
- [ ] 프론트 전용 타입은 서버 타입에서 파생한다 (Pick, Omit, Partial 등)
- [ ] 특정 필드 타입이 필요하면 Lookup Type으로 추출한다 (`Dto['field']`)
- [ ] 라이브러리 래핑 시 `ComponentProps<typeof X>` 또는 라이브러리 export 타입을 사용한다
- [ ] 하드코딩된 유니온 대신 `as const` + `typeof`로 상수에서 타입을 파생한다

### 예시

```typescript
// ❌ Bad - 타입 직접 정의 (원본과 동기화 깨짐)
type Status = 'pending' | 'approved' | 'rejected';

// ✅ Good - 상수에서 타입 파생
const STATUSES = ['pending', 'approved', 'rejected'] as const;
type Status = (typeof STATUSES)[number];
```

## API 타입

### orval 생성 타입 사용

```typescript
// Good - orval 생성 타입
import type { UserResponseDto } from '@/shared/api/model';

// Bad - 직접 정의
interface User {
  id: number;
  name: string;
}
```

### Lookup Type 활용

```typescript
// API 응답의 특정 필드 타입 추출
type UserId = UserResponseDto['id'];
type WorldMembers = WorldResponseDto['members'];
```

## 컴포넌트 Props 타입

### 라이브러리 컴포넌트 래핑 시

```typescript
import type { ButtonProps } from '@atelier/blue';

// ComponentProps 또는 라이브러리 타입 사용
interface CustomButtonProps extends ButtonProps {
  customLabel?: string;
}

// 또는
type CustomButtonProps = React.ComponentProps<typeof Button> & {
  customLabel?: string;
};
```

### 기존 타입 변형

```typescript
// Pick - 필요한 것만
type UserSummary = Pick<UserResponseDto, 'id' | 'name' | 'avatar'>;

// Omit - 특정 필드 제외
type UserWithoutPassword = Omit<UserResponseDto, 'password'>;

// Partial - 모두 선택적으로
type UserUpdateDto = Partial<UserResponseDto>;
```

## 타입 좁히기 (Type Guard)

### `any` 대신

```typescript
// Bad
const data: any = response.data;
console.log(data.name);

// Good
const isUser = (data: unknown): data is UserResponseDto => {
  return typeof data === 'object' && data !== null && 'id' in data;
};

if (isUser(response.data)) {
  console.log(response.data.name);
}
```

### `as` 대신

```typescript
// Bad
const user = response as UserResponseDto;

// Good - 타입 가드 사용
if (isUser(response)) {
  const user = response; // 자동으로 UserResponseDto 타입
}
```

### `!` (non-null assertion) 대신

```typescript
// Bad
const name = user!.name;

// Good - optional chaining + nullish coalescing
const name = user?.name ?? 'Unknown';

// Good - 조건문으로 좁히기
if (user) {
  const name = user.name;
}
```

## 모던 문법

| 패턴 | Bad | Good |
|------|-----|------|
| Nullish Coalescing | `value \|\| default` | `value ?? default` |
| 배열 포함 여부 | `arr.indexOf(x) !== -1` | `arr.includes(x)` |
| 마지막 요소 | `arr[arr.length - 1]` | `arr.at(-1)` |

```typescript
// Nullish Coalescing - falsy 값(0, '')도 유효한 경우
const count = data.count ?? 0;  // data.count가 0이면 0 유지

// 배열 포함 여부
if (allowedRoles.includes(userRole)) { ... }

// 마지막 요소
const lastMessage = messages.at(-1);
```

## Discriminated Unions

상태 관리는 `isLoading` 플래그가 아닌, `status` 유니온 필드로 분기.

```typescript
// Bad - 플래그 조합으로 상태 관리
interface State {
  isLoading: boolean;
  isError: boolean;
  data?: User;
  error?: Error;
}

// Good - Discriminated Union으로 상태 명확화
type State =
  | { status: 'idle' }
  | { status: 'loading' }
  | { status: 'success'; data: User }
  | { status: 'error'; error: Error };

// 사용 시 타입이 자동으로 좁혀짐
if (state.status === 'success') {
  console.log(state.data.name); // data가 확실히 존재
}
```

## as const & satisfies

상수 데이터는 `as const`로 리터럴 타입을 고정하고, `satisfies`로 타입 체크를 병행.

```typescript
// as const - 리터럴 타입 고정
const ROLES = ['admin', 'user', 'guest'] as const;
type Role = (typeof ROLES)[number]; // 'admin' | 'user' | 'guest'

// satisfies - 타입 체크하면서 리터럴 유지
const config = {
  apiUrl: 'https://api.example.com',
  timeout: 5000,
} as const satisfies { apiUrl: string; timeout: number };
// config.apiUrl의 타입: 'https://api.example.com' (리터럴 유지)
```

## Exhaustive Check

switch-case문의 default에서 `never` 타입으로 누락된 케이스 방어.

```typescript
type Status = 'pending' | 'approved' | 'rejected';

function handleStatus(status: Status) {
  switch (status) {
    case 'pending':
      return '대기 중';
    case 'approved':
      return '승인됨';
    case 'rejected':
      return '거절됨';
    default:
      // 새로운 Status가 추가되면 컴파일 에러 발생
      const _exhaustive: never = status;
      throw new Error(`Unknown status: ${_exhaustive}`);
  }
}
```

## Type Predicate

배열 `filter` 사용 시 `is Type` 함수를 사용하여 null 등을 안전하게 제거.

```typescript
// Bad - filter 후에도 타입이 (User | null)[]
const users = items.filter((item) => item !== null);

// Good - Type Predicate로 타입 좁히기
const isNotNull = <T>(value: T | null): value is T => value !== null;
const users = items.filter(isNotNull); // User[]
```

## export 함수 반환 타입 명시

외부로 노출되는 함수나 복잡한 로직은 반환 타입을 명시하여 API 경계를 명확히.

```typescript
// Bad - 반환 타입 추론에 의존
export function createUser(data: CreateUserDto) {
  return { ...data, id: generateId(), createdAt: new Date() };
}

// Good - 반환 타입 명시
export function createUser(data: CreateUserDto): User {
  return { ...data, id: generateId(), createdAt: new Date() };
}
```
