# Design Review 스킬

> 구현계획을 읽고 설계가 적절한지 검증한다. 코드 작성 전/중에 수행.

## 사용 시점

- 구현계획 작성 완료 후
- 커맨드 `3.설계검토`에서 호출

## 프로세스

### 1. 현황 파악

`.agent/개발진행/구현계획.md` + 연관 기획서 읽기 (이미 구현된 경우 변경 코드도 확인)

### 2. 자유 설계 비평

**체크박스를 채우려 하지 말고** 설계 의도를 자유롭게 비평한다.

#### 핵심 질문 (모든 설계에 자문)

- 이 설계가 **과한가? 부족한가?**
- **더 단순하게** 할 수 있는 방법은 없는가?
- 주니어 개발자가 **5분 안에 의도를 파악**할 수 있는가?
- **6개월 후의 나**도 이 설계를 이해하고 유지보수할 수 있는가?

#### 검토 관점

- **단순성**: 현재 요구사항에 필요한 것만 있는가? 미래를 위한 과잉 설계는 없는가?
- **일관성**: 기존 코드베이스의 패턴, 네이밍, 구조를 따르는가?
- **책임 분리**: 변경 이유가 오직 하나인가? UI와 도메인이 분리되었는가?
- **상태 관리**: 같은 데이터가 여러 곳에서 관리되진 않는가?
- **결합도**: 수정 시 연쇄 변경이 발생하는가?
- **순수성**: 핵심 로직이 외부 상태 없이 동작하는가? 부수효과가 격리되었는가?
- **확장성**: 요구사항 변경 시 수정 범위가 합리적인가? 한 파일만 고치면 되는가?
- **기획 정합성**: 기획서의 모든 요구사항이 반영되었는가?
- **실행 가능성**: AI가 이 프롬프트만 보고 오해 없이 작업할 수 있는가?

### 3. 프로젝트 규칙 확인

- [ ] **FSD 의존 방향**: 상위 → 하위만 import (`app/` → `pages/` → `features/` → `shared/`)
- [ ] **기존 패턴과 일관성**: 유사한 기존 페이지/컴포넌트가 있다면 그 구조를 따르는가?
- [ ] **shared/ui 순수성**: `shared/ui/`에서 API 호출이나 도메인 로직이 없는가?
- [ ] **Prop Drilling 금지**: 3단계 이상 props 내리지 않는가? (React Query 캐싱 활용)

필요 시 현재 코드베이스를 탐색하여 기존 패턴과의 일관성을 확인한다.

### 4. 결과 보고

#### 문제 없음

> "설계검토 완료. 문제 없습니다.
> **4.구현** 커맨드를 실행해주세요."

#### 개선 필요

1. 발견된 문제점과 개선 방향 정리
2. 사용자 승인
3. `.agent/개발진행/구현계획.md` 수정 (이미 구현된 코드의 설계를 수정하는 경우 기존 세션 건드리지 말고 리팩토링 세션 추가)
