# 상태 관리

## 데이터 호출 원칙 (Colocation)

### 핵심: 데이터가 필요한 컴포넌트가 직접 호출

- 상위 컴포넌트에서 Props로 내려주기를 **기다리지 않는다**
- 자식이 독립적으로 존재할 수 있다면, 자식이 직접 API 훅을 호출한다

### 중복 호출 허용

동일한 데이터가 A, B 두 컴포넌트에서 필요하면, **두 곳 모두에서 훅을 호출**한다.

- React Query가 캐싱으로 **네트워크 요청은 1회만** 발생
- 억지로 상위로 끌어올려서 Props로 내리면 **Prop Drilling** 발생
- 컴포넌트가 **독립적**이 되어 재사용성 향상

### 의사결정 기준 (Props vs Hook)

| 상황 | 방식 | 이유 |
|------|:---:|------|
| **`shared/ui/` 컴포넌트** | **Props** | 디자인 시스템은 비즈니스 로직(API)을 몰라야 함 |
| **리스트 아이템 (map 내부)** | **Props** | N개의 자식이 동시에 훅을 호출하는 오버헤드 방지 |
| **부모가 데이터를 가공해야 함** | **Props** | 필터링/정렬 등 부모의 UI 상태에 의존적인 경우 |
| **그 외 (독립적 기능)** | **Hook** | `Sidebar`, `Header`, `UserProfile` 등 독립적인 블록 |

## 상태 저장 위치 선택

| 상태 종류 | 도구 | 기준 |
|----------|------|------|
| 서버 데이터 | React Query | API 응답 |
| URL 공유 필요 | searchParams | 새로고침 했을 때 유지되어야 하는 정보들 |
| 로컬 상태 | useState | 1~2 depth, 해당 컴포넌트만 사용 |
| 그 외 공유 상태 | Zustand | 3 depth 이상 또는 여러 컴포넌트 공유 |

## 판단 기준

### Zustand 사용 기준

다음 중 하나에 해당하면 Zustand 사용:

1. **3 depth 이상 + 중간 컴포넌트가 해당 props를 안 쓰는 경우** (진짜 drilling)
2. **서로 다른 트리**의 컴포넌트가 같은 상태를 공유할 때

```
// 예시: 진짜 drilling (Zustand 사용)
Page → Layout → Sidebar → SidebarItem
         ↑ props를 그냥 전달만 함 (사용 안 함)

// 예시: drilling 아님 (useState로 충분)
Page → Form → Input
        ↑ props를 가공하거나 사용함
```

### Context / Jotai 사용 금지

- **새 코드**: Context, Jotai 사용 금지 → Zustand 사용
- **기존 코드**: 수정 시 Zustand로 마이그레이션
- 영향 범위가 크면 기술부채로 기록 후 별도 작업

### Zustand 리렌더 체크리스트

- [ ] 전체 스토어 구독 (`const store = useStore()`) 금지
- [ ] 값 가져올 때 selector 사용 (`useStore((s) => s.value)`)
- [ ] 객체로 여러 값 반환 시 `useShallow` 사용

```typescript
// Bad - 전체 스토어 구독 (불필요한 리렌더링)
const store = useAuthStore();
const userId = store.userId;

// Good - 단일 값은 selector
const userId = useAuthStore((s) => s.userId);

// Good - 여러 값은 useShallow
import { useShallow } from 'zustand/react/shallow';

const { isLoggedIn, logout } = useAuthStore(
  useShallow((s) => ({
    isLoggedIn: s.isLoggedIn,
    logout: s.logout,
  }))
);
```

## 데이터 페칭 전략

### 읽기 (Query)

| 상황 | 방식 |
|------|------|
| 단일 데이터 조회 | 일반 쿼리 (`useXxx`) + `isLoading`/`isError` 처리 |
| 코드 스플리팅, 복합 데이터 | Suspense 쿼리 사용 가능 |

```typescript
// 단일 데이터 - 일반 쿼리
const { data, isLoading, isError } = useUsersControllerGetMe();
if (isLoading) return <Skeleton />;
if (isError) return <ErrorMessage />;
```

### 쓰기 (Mutation)

```typescript
const [isSubmitting, setIsSubmitting] = useState(false);

const handleSubmit = async () => {
  setIsSubmitting(true);
  try {
    await usersControllerUpdate(data);
    message.success('저장되었습니다');
    queryClient.invalidateQueries({ queryKey: getUsersControllerGetMeQueryKey() });
  } catch {
    // 에러는 인터셉터에서 자동 처리
  } finally {
    setIsSubmitting(false);
  }
};
```

- orval 함수 직접 호출 + `useState`로 loading 관리
- 성공 후 invalidate할 Query 파악 필수

### Query Key 관리

Query Key는 orval이 생성한 함수 사용:

```typescript
import { getUsersControllerGetMeQueryKey } from '@/shared/api/generated';

// invalidate 시
queryClient.invalidateQueries({ queryKey: getUsersControllerGetMeQueryKey() });
```

### React Query 래핑

`useQuery` 직접 노출 대신 도메인 훅으로 감싸기:

```typescript
// Bad - 컴포넌트에서 useQuery 직접 사용
const MyComponent = () => {
  const { data } = useQuery({
    queryKey: getMcpControllerGetMcpQueryKey(mcpId),
    queryFn: () => mcpControllerGetMcp(mcpId),
  });
};

// Good - 도메인 훅으로 래핑
// features/mcp/hooks/useMcpData.ts
export const useMcpData = (mcpId: string) => {
  return useMcpControllerGetMcp(mcpId); // orval 생성 훅 사용
};
```

## 폼 관리

### 기본 원칙

| 상황 | 사용 도구 |
|------|----------|
| 간단한 폼 (필드 1-2개, 검증 없음) | `useState` |
| 복잡한 폼 (검증 필요, 필드 3개 이상) | `react-hook-form` |

### useState로 간단한 폼

```typescript
const [name, setName] = useState('');
const [email, setEmail] = useState('');

const handleSubmit = () => {
  mutate({ name, email });
};
```

### react-hook-form으로 복잡한 폼

검증이 필요하거나 필드가 많을 때:

```typescript
import { useForm } from 'react-hook-form';

interface FormData {
  name: string;
  email: string;
}

const { register, handleSubmit, formState: { errors } } = useForm<FormData>();

const onSubmit = (data: FormData) => {
  mutate(data);
};
```

### 수정 페이지 패턴

비동기 데이터 로딩 후 폼 채우기:

```typescript
const { data } = useUsersControllerGetMe();
const { register, reset } = useForm<FormData>();

useEffect(() => {
  if (data) {
    reset(data);  // react-hook-form의 reset으로 초기값 설정
  }
}, [data, reset]);
```

### Validate Mode 설정

UX에 맞게 검증 시점 선택:

| 상황 | mode |
|------|------|
| 즉각적 피드백 필요 | `onChange` |
| 제출 시에만 검증 | `onSubmit` |
| blur 시 검증 | `onBlur` |

```typescript
const { register } = useForm<FormData>({
  mode: 'onBlur',  // blur 시 검증
});
```

## 실시간 데이터 (WebSocket + React Query)

### 원칙: 서버 상태를 복사하지 말고, 캐시를 업데이트하라

React Query가 관리하는 서버 상태를 로컬 useState로 복사하면 **이중 상태(Dual Source of Truth)** 문제가 발생한다.

### 잘못된 패턴 (상태 복사 + useEffect 동기화)

```typescript
// 1. API에서 데이터 가져옴
const { data: logsFromApi } = useGetChatLogs(storyId);

// 2. 로컬 상태에 복사
const [logs, setLogs] = useState<ChatLogItemDto[]>([]);

// 3. useEffect로 동기화
useEffect(() => {
  if (logsFromApi) setLogs(logsFromApi);
}, [logsFromApi]);

// 4. WebSocket 이벤트 시 로컬 상태 업데이트
const handleNewLog = (log) => setLogs(prev => [...prev, log]);
```

**문제점:**
- **이중 상태**: React Query 캐시와 useState가 따로 존재
- **동기화 버그 위험**: useEffect 의존성 누락, 타이밍 이슈
- **메모리 낭비**: 같은 데이터가 두 곳에 저장됨
- **불일치 가능성**: 로컬 상태와 캐시가 달라질 수 있음

### 올바른 패턴 (React Query 캐시 직접 업데이트)

```typescript
const queryClient = useQueryClient();
const logsQueryKey = getLogsQueryKey(storyId, { limit: 100 });

// 1. API에서 데이터 가져옴 (캐시에 자동 저장)
const { data: logsData } = useGetChatLogs(storyId, { limit: 100 });
const logs = logsData?.data?.logs ?? [];

// 2. WebSocket 이벤트 시 캐시 직접 업데이트
const handleNewLog = useCallback((log: ChatLogItemDto) => {
  queryClient.setQueryData(logsQueryKey, (old) => {
    if (!old?.data?.logs) return old;
    if (old.data.logs.some((l) => l.id === log.id)) return old;
    return {
      ...old,
      data: { ...old.data, logs: [...old.data.logs, log] }
    };
  });
}, [queryClient, logsQueryKey]);
```

**장점:**

| 항목 | 설명 |
|------|------|
| **Single Source of Truth** | React Query 캐시만 사용, 상태가 한 곳에만 존재 |
| **자동 구독** | `setQueryData` 호출 시 해당 쿼리를 구독하는 모든 컴포넌트가 자동 리렌더 |
| **useEffect 제거** | 동기화 로직 불필요, 버그 가능성 감소 |
| **캐시 무효화와 통합** | `invalidateQueries`와 일관된 데이터 흐름 |

### 예외: 로컬 상태가 필요한 경우

| 상황 | 예시 | 이유 |
|------|------|------|
| **편집 중인 임시 데이터** | 폼 입력값, 스타일 편집 | 저장 전까지 서버와 무관 |
| **UI 전용 상태** | 모달 열림, 입력 타입 | 서버에 저장할 필요 없음 |
| **낙관적 업데이트 롤백용** | 이전 상태 백업 | 에러 시 복원 필요 |
