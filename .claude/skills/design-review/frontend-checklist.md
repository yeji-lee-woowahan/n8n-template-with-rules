## FSD 구조 검토

### 레이어 배치

- [ ] 새 컴포넌트/훅/유틸의 **배치 위치가 적절**한가?

| 질문 | 배치 위치 |
|------|----------|
| 앱 전역에서 쓰이는 공용 컴포넌트? | `shared/ui/` |
| 특정 도메인 기능? | `features/{domain}/` |
| 페이지 진입점? | `pages/` |
| 앱 설정/프로바이더? | `app/` |

### 의존 방향

- [ ] **상위 → 하위만 import** 하는가? (역방향 금지)
  - `app/` → `pages/` → `features/` → `shared/`
- [ ] `shared/`에서 `features/`를 import하지 않는가?
- [ ] `features/A`에서 `features/B`를 직접 import하지 않는가?

### features/ 분류

- [ ] **"무엇을 다루는가"** 기준으로 분류했는가? (사용되는 곳 기준 X)
  - 예: MySpace 페이지에서 쓰는 MCP 관련 모달 → `features/mcp/`

### 레이아웃 배치 판단

- [ ] `app/layouts/` vs `shared/ui/` 배치가 적절한가?

| 구분 | `app/layouts/` | `shared/ui/` |
|------|----------------|--------------|
| 역할 | **앱 Shell** (네비게이션 구조) | **순수 UI 컴포넌트** |
| 포함 내용 | 헤더, 사이드바, 탭바, 라우팅 로직 | 스타일링만 (로직 없음) |
| 크기 | 대형 (300줄+) | 소형 (50줄 미만) |
| 예시 | `MainLayout` | `PageWrapper`, `ContentCard` |

→ 상세: [FSD-폴더구조.md](./FSD-폴더구조.md)

## 컴포넌트 설계

### UI vs 도메인 컴포넌트 분리

- [ ] **UI 컴포넌트** (`shared/ui/`)는 API를 호출하지 않는가?
- [ ] **도메인 컴포넌트** (`features/{domain}/components/`)가 API 호출 + UI 조합을 담당하는가?
- [ ] 컴포넌트 이름에 **비즈니스 맥락**이 포함되어 있는가? (API 호출하는 경우)

### 분리 전략

- [ ] **분리가 정말 필요한가?** 다음 중 하나 이상 충족해야 분리:
  - 순수 재사용 (비즈니스 로직은 다르지만, 똑같은 디자인이 다른 곳에서 필요)
  - 공통 UI 라이브러리 (도메인 맥락 없이 프로젝트 전반에서 사용)
- [ ] **과한 추상화**를 하지 않았는가?
  - 한 곳에서만 쓰이는데 Container/View 분리 X
  - 기본은 로직+UI를 한 파일에 작성

→ 상세: [컴포넌트-설계.md](./컴포넌트-설계.md)

## 상태 관리 설계

### 상태 저장 위치

- [ ] **상태 저장 위치가 적절**한가?

| 데이터 종류 | 사용 도구 |
|------------|----------|
| API 서버 데이터 | React Query |
| 전역 클라이언트 상태 | Zustand |
| 공유 가능해야 하는 상태 (탭, 필터, 페이지네이션) | URL |
| 컴포넌트 로컬 상태 | useState |

### 상태 관리 원칙

- [ ] **단일 진실의 원천**: 상태가 한 곳에서만 관리되는가? (외부/내부 혼용 X)
- [ ] **상태 가까이 두기**: 상태가 사용되는 곳 가까이에 있는가?
- [ ] **캡슐화**: 커스텀 훅 + 컴포넌트 조합에서 상태는 훅 안에 있는가?
- [ ] **Context/Jotai 사용 금지**: 새 코드에서 Context, Jotai 사용 X → Zustand 사용
- [ ] **Zustand selector 필수**: 전체 스토어 구독 금지, selector 또는 useShallow 사용

### 데이터 호출 원칙 (Colocation)

- [ ] **Prop Drilling 하지 않는가?** 자식이 직접 API 훅을 호출할 수 있는데, 부모가 Props로 3단계 이상 내려주고 있지 않은가?
- [ ] **중복 호출을 두려워하지 않는가?** 서로 멀리 떨어진 컴포넌트에서 같은 데이터 필요 시, 각자 훅을 호출 (React Query 캐싱)
- [ ] **반복문 내부 호출 금지**: 리스트 아이템(map 내부)에서는 API 훅 호출 X

→ 상세: [상태관리.md](./상태관리.md)

## 커스텀 훅 설계

- [ ] **Headless 원칙**: 훅에서 JSX 반환하지 않는가?
- [ ] **훅이 상태를 소유**하고, 컴포넌트는 UI만 담당하는가?
- [ ] 외부 prop은 **트리거/힌트 역할만**, 훅 내부 상태에 동기화하는가?
- [ ] 반환값 3개 이상이면 **객체로 반환**하는가?
- [ ] 거대한 `usePageLogic` 대신 **작은 훅들을 조합**하는가?

## 설계 원칙 (구조)

| 원칙 | 질문 |
|------|------|
| **SRP** | 컴포넌트/훅/함수의 변경 이유가 오직 하나인가? |
| **OCP** | if-else 분기 대신 props/hook 주입으로 확장 가능한가? |
| **ISP** | 컴포넌트에 필요 없는 Props까지 통째로 넘기지 않는가? |
| **DIP** | 라이브러리(axios 등)에 직접 의존하지 않고 훅으로 추상화했는가? |
| **DI** | 테스트 시 외부 의존성을 쉽게 교체할 수 있는가? (훅/props 주입) |

### if-else 분기

## 상수 위치 결정

- [ ] **기획 값** (최대 글자수, 재시도 횟수, 유효기간) → `shared/constants`
- [ ] **글로벌 영향 값** (z-index, Timeout, 브레이크포인트) → `shared/constants`
- [ ] **구현 상세** (애니메이션 duration, 디바운스 시간) → 해당 파일 상단
