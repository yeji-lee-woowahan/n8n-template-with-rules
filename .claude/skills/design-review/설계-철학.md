# 설계 철학

> 설계 단계에서 지켜야 할 원칙과 검증 기준

---

## 오버 엔지니어링 필터

설계 전 반드시 자문:

1. **이 설계가 해결하는 문제가 실제로 존재하는가?**
2. **코드 복잡도 증가에 상응하는 측정 가능한 이득이 있는가?**
3. **주니어 개발자가 5분 안에 의도를 파악할 수 있는가?**
4. **현재 요구사항에 정확히 필요한 것만 설계했는가? (YAGNI)**

---

## 일관성 원칙

- **유사한 기존 페이지가 있다면, 그 구조와 패턴을 그대로 승계**
- 기존 패턴과 다른 방식을 사용한다면, **왜 불가능한지 설명 가능해야 함**
- 파일명, 컴포넌트명, 폴더 구조가 **기존과 동일한 규칙** 따르기

---

## 구조적 한계 대응

기존 패턴으로 해결 안 될 때:

| 선택 | 설명 |
|------|------|
| **일반화** | 영향받는 모든 파일 파악 → 스코프에 포함 |
| **격리** | `// 예외 처리: [이유]` 주석 + 문서 기록 |

둘 중 하나 선택. 애매하게 섞지 않기.

## 코드 품질

### 하드코딩 vs 리터럴

**상수화 필요:**
| 항목 | Bad | Good |
|------|-----|------|
| 비즈니스 의미 있는 숫자 | `if (count > 100)` | `if (count > MAX_ITEMS)` |
| 여러 곳에서 쓰는 문자열 | `role === 'admin'` | `role === ROLES.ADMIN` |
| 변경 가능성 있는 값 | `setTimeout(..., 3000)` | `setTimeout(..., RETRY_DELAY)` |

**리터럴 허용:**
| 항목 | 예시 | 이유 |
|------|------|------|
| 단순 UI 수치 | `padding: 16`, `gap: 8` | 짝수 권장 |
| 배열 인덱스 | `items[0]`, `arr.slice(0, 3)` | 의미가 명확함 |
| 한 곳에서만 쓰는 값 | `opacity: 0.8` | 상수화해도 이득 없음 |
| 수학적 상수 | `/ 2`, `* 100` | 보편적 의미 |

### 상수 위치 결정

**shared/constants에 두는 경우:**

| 종류 | 예시 | 이유 |
|------|------|------|
| 기획 값 | 최대 글자수, 재시도 횟수, 유효기간 | 기획 변경 시 한 곳만 수정 |
| 글로벌 영향 값 | z-index, Timeout, 브레이크포인트 | 여러 파일에서 참조, 충돌 방지 |

```typescript
// shared/constants/
export const MAX_MCP_NAME_LENGTH = 50;     // 기획 값
export const DEFAULT_PAGE_SIZE = 20;       // 기획 값
export const Z_INDEX = { MODAL: 1000, TOAST: 1100 };  // 글로벌 영향
```

**파일 상단에 두는 경우 (Colocation):**

| 종류 | 예시 | 이유 |
|------|------|------|
| 구현 상세 | 애니메이션 duration, 디바운스 시간 | 해당 컴포넌트에서만 사용 |
| UI 임계값 | 스크롤 트리거 위치, 리사이즈 기준 | 컴포넌트 로직과 밀접 |

```typescript
// SomeComponent.tsx 상단
const ANIMATION_DURATION = 200;
const DEBOUNCE_MS = 300;
const SCROLL_THRESHOLD = 100;

export function SomeComponent() { ... }
```

**판단 기준:**
- "기획서에 명시된 값인가?" → shared/constants
- "이 파일에서만 쓰는 구현 디테일인가?" → 파일 상단
- "다른 파일에서도 같은 값을 참조해야 하는가?" → shared/constants

## 설계 원칙 (구조)

| 원칙 | 질문 |
|------|------|
| **SRP** | 컴포넌트/훅/함수의 변경 이유가 오직 하나인가? |
| **OCP** | if-else 분기 대신 props/hook 주입으로 확장 가능한가? |
| **ISP** | 컴포넌트에 필요 없는 Props까지 통째로 넘기지 않는가? |
| **DIP** | 라이브러리(axios 등)에 직접 의존하지 않고 훅으로 추상화했는가? |

### if-else 제거

3개 이상 분기 시 객체 맵 사용:

```typescript
// Bad
const getStatusText = (status: string) => {
  if (status === 'pending') return '대기중';
  else if (status === 'active') return '활성';
  else if (status === 'inactive') return '비활성';
  else return '알 수 없음';
};

// Good
const STATUS_TEXT: Record<string, string> = {
  pending: '대기중',
  active: '활성',
  inactive: '비활성',
};

const getStatusText = (status: string) => STATUS_TEXT[status] ?? '알 수 없음';
```

| 원칙 | 질문 |
|------|------|
| **순수 함수** | 핵심 로직이 외부 상태 없이 동작하는가? |
| **불변성** | 객체/배열 수정 시 새 객체를 반환하는가? |
| **부수효과 격리** | API 호출, DOM 조작은 핸들러나 `useEffect`에만 존재하는가? |
