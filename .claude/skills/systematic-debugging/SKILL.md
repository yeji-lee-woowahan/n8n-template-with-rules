# Systematic Debugging 스킬

> 버그를 보면 "왜 이게 발생했지?"를 역추적하며 원인의 범위를 좁혀나간다.
> 가설을 세우고 검증하며, 섣불리 수정하기보다 원인을 먼저 확정한다.
> 수정할 때는 기존 기능을 깨뜨리지 않는 방향을 항상 염두에 둔다.

## 사용 시점

- 버그 수정 요청 시
- 커맨드 `1.버그수정`에서 호출

## 핵심 원칙

```
근본 원인 조사 없이 수정 금지
```

랜덤 수정은 시간 낭비 + 새 버그를 만든다. 증상만 고치는 것은 실패다.

## 프로세스

### 1. 이슈 및 기획 파악

사용자에게 해결하고자 하는 이슈를 듣고 문제 발생 부분의 기획을 파악한다.

- 기획서(`n8n-template/docs/기획서/`)에서 관련 문서 찾기
- "필수 확인 동작" 섹션이 있으면 꼼꼼하게 읽기

### 2. 근본 원인 조사

**수정 시도 전에 반드시:**

1. **에러 메시지를 꼼꼼히 읽는다**
   - 스택 트레이스 전체 확인
   - 파일 경로, 라인 번호, 에러 코드 기록
   - 건너뛰지 않는다

2. **재현을 확인한다**
   - 매번 발생하는가?
   - 정확한 재현 단계는?
   - 재현 불가하면 추측하지 말고 데이터를 더 모은다

3. **최근 변경을 확인한다**
   - `git diff`, 최근 커밋
   - 새 의존성, 설정 변경
   - 환경 차이

4. **데이터 흐름을 추적한다**
   - 잘못된 값이 어디서 시작되는가?
   - 누가 이 값을 전달했는가?
   - 소스를 찾을 때까지 역추적
   - **증상이 아닌 소스에서 수정**

### 3. 패턴 분석

1. **동작하는 유사 코드를 찾는다**
   - 같은 코드베이스에서 비슷하게 동작하는 것
2. **차이점을 식별한다**
   - 동작하는 것과 깨진 것의 모든 차이를 나열
   - "이건 상관없을 거야"라고 가정하지 않는다
3. **의존성을 파악한다**
   - 다른 컴포넌트/설정/환경이 필요한가?

### 4. 가설 검증

1. **단일 가설 수립**: "X가 원인이다. 왜냐하면 Y"
2. **최소한의 변경으로 테스트**: 한 번에 하나만 변경
3. **검증 후 진행**:
   - 성공 → 5단계로
   - 실패 → 새 가설 수립 (기존 수정 위에 덧붙이지 않는다)

### 5. 구현

1. **실패하는 테스트 먼저 작성** (가능하면)
2. **단일 수정 적용**: 근본 원인만 해결, "겸사겸사" 리팩토링 금지
3. **검증**: 테스트 통과 + 다른 테스트 깨지지 않음

### 3회 이상 수정 실패 시

**즉시 중단하고 아키텍처를 의심한다:**
- 매번 수정할 때마다 다른 곳에서 새 문제가 나타나는가?
- "대규모 리팩토링" 없이는 고칠 수 없는가?
- 각 수정이 다른 곳에 사이드이펙트를 만드는가?

→ 더 많은 수정 시도 대신 **사용자와 근본적 설계 변경을 논의**

### 6. 기획서 및 테스트 업데이트

- 기획이 수정되어야 하는 부분은 기획서 수정
  - `n8n-template/docs/기획서/0-기획서-작성가이드.md` 참고
- 한번 발생했던 이슈가 재발하지 않도록 "필수 확인 동작" 섹션 꼼꼼하게 작성
- 수정 전 사용자에게 확인받고 진행

### 7. 구현계획 정리

같은 세션에서 `2.구현계획` 커맨드 실행을 요청한다.

## 디버깅 체크리스트

원인이 불명확할 때 순차적으로 확인:

1. **재현 확인**: 동일 조건에서 재현되는가?
2. **범위 축소**: 어떤 컴포넌트/훅에서 발생하는가?
3. **상태 추적**: 관련 상태의 초기값 → 변경 시점 → 최종값
4. **이벤트 순서**: 사용자 액션 → 핸들러 → 상태 변경 → 리렌더 순서
5. **의존성 확인**: useEffect/useMemo 의존성 배열이 올바른가?
6. **타이밍 이슈**: 비동기 호출, 디바운스, 애니메이션 타이밍
7. **캐시 문제**: React Query 캐시가 stale 데이터를 보여주는가?

## 위험 신호 - 즉시 멈추고 2단계로

이런 생각이 들면 프로세스를 따르지 않고 있는 것:

- "일단 빠르게 고치고 나중에 조사하자"
- "X를 바꿔보면 될 것 같은데"
- "여러 개 한꺼번에 고치고 테스트하자"
- "아마 X일 거야, 고쳐보자"
- "완전히 이해한 건 아니지만 이렇게 하면 될 수도"
- "한 번만 더 시도해보자" (이미 2회 이상 실패한 경우)

## 빠른 참조

| 단계 | 핵심 활동 | 완료 기준 |
|------|-----------|-----------|
| **1. 이슈 파악** | 기획서 확인, 문제 이해 | 이슈와 기획 관계 파악 |
| **2. 근본 원인** | 에러 읽기, 재현, 변경 확인, 추적 | 무엇이 왜 깨졌는지 이해 |
| **3. 패턴** | 동작하는 예시 찾기, 차이 비교 | 차이점 식별 완료 |
| **4. 가설** | 이론 수립, 최소 변경으로 검증 | 가설 확인 또는 새 가설 |
| **5. 구현** | 테스트 작성, 수정, 검증 | 버그 해결 + 테스트 통과 |
