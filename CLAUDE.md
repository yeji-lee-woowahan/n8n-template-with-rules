# 프로젝트 규칙

## 프로젝트 구조

- 코드: `n8n-template/src/**`
- 기획서: `n8n-template/docs/기획서/`
- 구현계획: `.agent/개발진행/구현계획.md`
- 스킬 상세 문서: `.claude/skills/` (참조용)

## 절대 규칙

- 파일 이름/위치 변경 시 `git mv` 또는 `mv` 사용 (파일 새로 생성 후 삭제 금지)
- 파일 분리 작업 시 새 이름/폴더에 작업 후 기존과 비교할 수 있게 유지
- 사용자의 명시적 요청 없이는 dev server를 실행하지 않는다
- `.claude/` 또는 `.cursor/` 내 파일(rules, skills, commands)을 변경할 때, 반대쪽 폴더에도 동일한 변경을 즉시 동기화한다
- "~는 없습니다/지원하지 않습니다/불가능합니다" 등 도구·플랫폼 기능에 대한 부정형 단정을 하려면 → **반드시 WebSearch로 최신 문서 확인 후 답변** (검색 안 했으면 "(훈련 데이터 기준, 미검증)" 명시)

## 수정 전 확인

수정을 진행하기 전에는 항상:

1. **변경 계획을 먼저 요약**해서 보여준다
2. 확인 후 진행한다

요청 표현과 관계없이, 수정이 필요해 보이면 먼저 제안하고 확인받는다.

### 변경 방안 보고 형식

코드·기획서·설계 등 수정이 필요한 경우:

1. **가장 간단한 해결방법** — 최소 변경으로 문제를 해소하는 방법
2. **근본적인 해결방법** — 원인을 제거하는 방법 (백지상태에서 설계한다면 어떻게 했을지)
3. **제안하는 방법** — 위 두 가지를 고려한 종합 판단 (근거 포함)

각 방안에 함께 기재:
- **영향받는 부분** — 변경으로 인해 영향받는 파일·컴포넌트·기능
- **추가 고려사항** — 트레이드오프, 후속 작업, 주의할 점

> 영향 범위가 작고 명확한 경우 간략히 기재해도 된다.

## 탐색 제한

- 코드 검색 시 `n8n-template/` 폴더 내에서만 검색
- 루트의 설정 파일이나 `.agent/` 폴더는 필요시에만 참조

## Git 환경

`n8n-template/`은 바깥 폴더(`n8n-template-with-rules/`)와 별도의 독립 git 저장소다.

- **모든 git 명령**은 `/Users/yeji.lee/dev/n8n-template-with-rules/n8n-template` 에서 실행
- 커밋·PR 생성 시 `n8n-template/` 내부의 git을 사용

## 위키 URL 접근

`cloud.wiki.woowa.in` 도메인의 URL은 직접 fetch할 수 없다.
반드시 MCP의 `confluence_get_page`를 사용한다.

URL 형식: `https://cloud.wiki.woowa.in/wiki/spaces/{SPACE_KEY}/pages/{PAGE_ID}/...`
- `pageId`: URL의 `pages/` 뒤 숫자 (예: `993139815`)

## 코드 검증

- 구현 완료 후 타입 확인: `pnpm typecheck` (n8n-template/ 에서 실행)
- 커밋 전 최종 검증: `pnpm verify` (ESLint + Prettier + TypeScript)

## 버그 수정 원칙

- **근본 원인 조사 없이 수정 금지** — 랜덤 수정은 새 버그를 만든다
- 에러 메시지를 꼼꼼히 읽고, 데이터 흐름을 추적하여 소스에서 수정
- 3회 이상 수정 실패 시 즉시 중단하고 사용자와 근본적 설계 변경 논의
- 상세 프로세스: `.claude/skills/systematic-debugging/SKILL.md`

## 워크플로우 권고

ad-hoc 작업이 이어질 때, 아래 조건에 해당하면 다음 커맨드 실행을 권고한다.

| 상황 | 권고 커맨드 |
|------|------------|
| 구현계획.md에 내용이 있고, 코드 변경 없이 논의만 진행 중 | `/구현` |
| 코드를 여러 차례 수정했으나 검증을 안 함 | `/가상검증` 또는 `/코드리뷰` |
| 기능 구현이 마무리 단계 | `/비판적검토` → `/문서정리` → `/커밋` |
| 버그를 ad-hoc으로 고치고 있으나 원인이 복잡함 | `/버그수정` (체계적 디버깅) |
| 새 기능 논의 중인데 기획서가 없음 | `/기능추가` |
| 코드 정리가 필요해 보이는 상태 | `/코드정리` |
| 구현계획 없이 큰 작업을 시작하려 함 | `/구현계획` |

권고 형식: 대화 끝에 아래 형태로 커맨드와 함께 사용할 프롬프트를 작성한다.

```
> 다음 단계로 **/X커맨드** 실행을 권합니다 — (이유)
>
> **프롬프트:**
> (커맨드 실행 시 입력할 프롬프트 내용)
```
